What building a lock-free queue taught me about thinking at CPU timescales

I built an experimental worker pool in Go — not to ship it, but to understand what happens below the goroutine. Here's what three different machines showed me.

Andrey Azargarov  ·  2026  ·  github.com/azargarov/wpool
01 / 06

How this project changed direction

I started this project with a simple goal: build a worker pool in Go and make it faster than the straightforward versions I already knew. The first versions were conventional and practical, based on channels, mutexes, and a fixed group of workers. At that stage I was mostly driven by curiosity about performance. I wanted to see whether I could push the design further and build something faster than a simple pool.

I was not thinking much about hardware at the beginning. My focus was on the queue itself and on making the design work. The questions about CPU caches, memory locality, scheduling, and machine topology came later, when the implementation became more complex and some of the bottlenecks were no longer visible at the language level.

That is how the project evolved into a lock-free segmented queue design. At first the change was mostly driven by performance. I wanted to see whether a different queue structure could push the pool further than the simpler channel-and-mutex versions. Only later, while trying to make the design work correctly and consistently under load, did I start paying attention to what was happening underneath: contention on atomic operations, cache behavior, memory locality, and machine topology. Building the queue gradually forced me to think at a lower level than I had expected at the beginning.

One of the more useful lessons was that correctness in concurrent code is shaped by the runtime as much as by the algorithm. During development I ran into cases where a goroutine could be delayed long enough to make segment reuse unsafe if jobs were still referenced from shared queue memory. That led me to copy claimed jobs into a private slice before execution and to add a simple limbo stage before segment reuse. I know there are more established reclamation techniques, but in this project I wanted to solve the problem in a way that made the lifetime issues visible and understandable to me.

At that point the goal of the project changed. I realized that the more valuable result was the investigation itself. Maximum throughput stopped being the only goal. The project became a way to study lock-free design in practice.

02 / 06

The design: why lock-free doesn't mean free

A standard channel-based pool serializes submission through a mutex inside the runtime. The problem is that mutex forces sequential execution of the queue itself. When jobs are short and cheap, producers spend more time waiting to acquire the lock than actually submitting work. Add to that the overhead of parking and unparking OS threads on every lock/unlock cycle, and the potential parallelism of the system is never realised. More producers simply means a longer queue at the mutex door.

A lock-free queue replaces the mutex with atomic CAS operations (Compare-And-Swap). Each producer reserves a slot independently. If two producers collide, one retries — but neither blocks, and neither parks an OS thread. The queue is divided into fixed-size segments. Each segment contains two arrays: a jobs[] array holding the actual work, and a ready[] array used for slot reservation.

The ready[] array stores the current epoch of the segment, not a simple boolean flag. When a segment is recycled for a new generation, the epoch increments — so there is no need to zero out the array between cycles. A slot is considered available when its stored epoch matches the current segment epoch; once a producer writes a job and stamps the slot with the current epoch, the consumer can see it as ready. This avoids a full memset on every segment recycle, which matters when segments are large.

The hot path

// Producer: reserve one slot, write job, then publish it.
idx := atomic.AddUint64(&seg.tail, 1) - 1
slot := idx % segSize

seg.jobs[slot] = job
atomic.StoreUint64(&seg.ready[slot], seg.epoch) // publish for this epoch

// Consumer: scan for a contiguous published run, then claim it by advancing head.
head := atomic.LoadUint64(&seg.head)
end := head

for end < atomic.LoadUint64(&seg.tail) {
    slot := end % segSize
    if atomic.LoadUint64(&seg.ready[slot]) != seg.epoch {
        break // reserved but not yet published
    }
    end++
}

if end > head && atomic.CompareAndSwapUint64(&seg.head, head, end) {
    count := end - head
    batch := make([]Job, count)

    // Copy claimed jobs out of the segment before execution.
    for i := uint64(0); i < count; i++ {
        batch[i] = seg.jobs[(head+i)%segSize]
    }

    seg.release() // segment can re-enter limbo before jobs finish running

    for _, job := range batch {
        execute(job)
    }
}

Consumers do not execute directly from segment memory. They first scan for a contiguous published run, claim that range by advancing the consumer head, copy the claimed jobs into a private batch, and only then execute them. This separates job execution lifetime from segment lifetime.

The copy-before-execute pattern is a direct response to a problem discovered during development: goroutines can sleep for a surprisingly long time while holding references to data associated with a live segment. Without the copy, reclamation stalls behind slow or descheduled workers. Copying the claimed range first lets the segment re-enter circulation earlier, which makes recycling more stable under real workloads.

The deeper insight is that locking and memory latency are different problems. Removing locks does not necessarily remove the cost of memory access. If the structure creates cache misses, that cost remains and may become the dominant one. Measuring that effect became an important part of this work.

⚠ Known Limitation

In my design, memory reclamation uses a heuristic "limbo" approach — segments that may still be referenced are deferred. With very small segments and a small pool, limbo can exhaust available segments under load. Proper epoch-based or hazard-pointer reclamation is still in progress.

What perf counters reveal that benchmarks hide

Throughput numbers tell how fast. Perf counters tell why. I ran the same pool against three different workloads — an empty no-op job, a SHA-256 hash, and a CPU-bound loop — while collecting hardware counters on the AMD Ryzen 7 8845HS system.

IPC — instructions per cycle — measures how much useful work the CPU does per clock tick. A healthy tight loop sits around 2–4. At 0.13 IPC for the empty job on the AMD, the CPU is almost entirely idle, waiting for memory.

Hardware counter comparison — three workloads, AMD Ryzen 7 8845HS (pinned, single socket)
All values normalised to 0–100 scale for comparison. Raw: IPC 0.13/1.18/1.50 · Cache miss% 38.8/29.5/26.2 · Frontend stall% 4.2/0.3/46.2

The cache miss rate is 38.8% for the empty job and falls to 26.2% for cpu-bound — the job's own working set fills the cache, reducing relative pressure from the queue. The 46% frontend stall for cpu-bound is entirely inside the job itself, a branch-heavy compute loop. The queue is invisible.

The cross-machine cache miss story is where it gets interesting. On the Xeon Gold with 40 workers pinned to one socket, the empty-job cache miss rate is 0.104% — more than two orders of magnitude lower than the AMD. The segment pool fits almost entirely in the shared L3 and stays hot. Switch to 80 workers free-scheduled across both sockets and it jumps to 37.2% — very close to the AMD result. The same effect appears for SHA-256: pinned gives 1.79% cache misses, while free scheduling raises that to 46.4%. For the CPU-bound workload, the miss rate also rises sharply, from 1.70% to 46.7%. That pattern suggests the cost is not only in the jobs themselves, but in cache-line movement caused by many workers coordinating across the NUMA boundary.

Queue overhead as fraction of total job cost
Queue overhead baseline = empty job with 1 producer / 1 consumer (~93ns/op). At 16p/16c the empty-job cost rises to ~155ns due to CAS contention between producers — so the "queue tax" on sha256 is even lower than the chart suggests, since heavier workloads run with the same contention that inflates the empty-job number.

“Lock-free is not free. It removes lock contention, but shifts cost into atomic coordination, memory traffic, cache behavior, and reclamation. Whether that trade pays off depends on the workload and the machine.”

One important nuance: the "queue overhead" baseline in the chart is measured with a single producer and single consumer — the minimum contention case, ~93ns/op. At 16 producers and 16 consumers pushing empty jobs, the measured cost rises to ~155ns because producers compete on CAS reservations. For heavier workloads like SHA-256, the real queue overhead fraction is therefore even smaller than shown — the empty-job number used as a baseline already includes contention that isn't present at the same scale when the job itself takes hundreds of nanoseconds to execute. To make a fair claim about queue overhead you need to measure job execution time in isolation, then subtract.

Finding 1: startup effects dominate early latency

The first job costs 50–200µs depending on the machine. By 10,000 jobs, p50 has dropped by 10–35×. The first job is not a clean queue measurement but mostly reflects OS thread scheduling, goroutine stack allocation, and TLB misses as segment-pool pages fault in.

A single average benchmark number hides that behavior. Under startup load, the first hundred requests absorb the cold-start tax.

Finding 2: the E5-2640 punches above its weight

The 2013 E5-2640 at 2.5GHz achieves p50 = 2.5µs and p99 = 68µs. Compare that to the Xeon Gold at 2.0GHz with 20 threads on one socket: p50 = 9µs, p99 = 102µs. The Xeon E5-2640 has better tail latency.

One likely explanation is that fewer cores and a simpler memory topology reduce contention around the segment pool and limbo reclamation path. If so, the limiting factor here is not just clock speed, but how the algorithm interacts with concurrency and topology.

Finding 3: cross-socket scheduling worsens tail latency

When workers are pinned to a single socket — regardless of which socket — p99 is ~380µs and throughput is ~680 ns/op. Both sockets are symmetric. When the OS schedules freely across both sockets, p99 jumps to ~2300µs: ~6× worse. The throughput paradox also appears: 80 free-scheduled threads produce worse ns/op (1165) than 40 pinned threads (680).

The perf counters strongly suggest the mechanism. With 40 threads pinned to a single socket, the empty-job cache miss rate is 0.104%. With 80 threads free-scheduled across both sockets, it rises to 37.2% — roughly 350× higher. That points less to the jobs themselves being limited by remote memory latency, and more to CAS contention from 80 goroutines spread across two sockets repeatedly touching the same queue structures across a coherence boundary. Under the limbo heuristic, occasional reclamation stalls appear to amplify that effect into millisecond-range tail spikes. In this setup, the practical fix is not to avoid NUMA entirely, but to pin workers to one socket so contention stays within a single coherence domain.

Segment size: configuration is architecture

The segment size is not just a tuning knob — it determines how the queue amortises per-operation overhead, and at the extreme it breaks the zero-allocation guarantee entirely.

I swept segment size from 1 to 4096 on all three machines with SEGCOUNT=128 constant. The shape of the curve differs by machine, and that difference tells a story about cache hierarchy.

Three findings stand out. First, size=1 breaks the zero-alloc guarantee on every machine — 2 allocs/op appear as limbo exhaustion forces the allocator to create new segments. On the Xeon GOLD 6138×2 with 80 workers this is catastrophic: 5839 ns/op vs 166 ns/op on the AMD 8845HS, because 80 concurrent goroutines exhaust the limbo queue far faster.

Second, the Xeon E5-2640 plateau appears at size=256 while AMD 8845HS and Xeon GOLD 6138×2 continue improving to 4096. The Xeon E5-2640 has a smaller L3 relative to the number of active segments — once segments exceed the cache working set, larger segments start evicting each other and the gains reverse. This is the L3 cache size written in benchmark numbers.

Third, bytes/op is non-zero even at 0 allocs — 33–41 bytes depending on size. This is stack growth and goroutine scheduler overhead, not heap allocation. Safe to ignore as long as allocs/op is zero.

What this project gave me

The measurements were useful because they gave me something more concrete than “this machine feels slower.” On some systems the queue behaved predictably and stayed stable once warm. On others, especially across sockets, the same design showed much worse tail behavior. That made the project less about proving one queue is better than another and more about learning how software structure meets hardware reality. The important result for me was not a single benchmark number, but a better ability to connect symptoms to mechanisms.

So I do not present this project as a finished answer. I see it more as a research step. It taught me that lock-free does not remove complexity; it moves complexity into places that are easier to ignore at first: memory visibility, cache movement, reclamation, and scheduling behavior. It also taught me that if I want to keep building concurrent structures, I need more theory, especially around safe memory reclamation and the broader design space of lock-free queues. That is probably the next stage of the work.

Even in its current state, though, the project was worth doing. It gave me a more practical understanding of what contention means, what cache effects look like in measurements, and why the same algorithm can behave very differently on different machines. That was the real outcome I was looking for, even if I only understood that clearly near the end.

The diagnostic ladder

One practical result of this project is that it changed the order in which I look at performance problems. When something is slow, this is usually where I start:

# Step 1: Is the CPU doing useful work?
perf stat -e cycles,instructions ...
# Very low IPC often points to stalls or memory pressure.
# Higher IPC usually means the workload is spending more time doing actual computation.

# Step 2: Where is memory pressure coming from?
perf stat -e cache-references,cache-misses ...
# A high miss rate suggests the hot working set is not staying in cache.

# Step 3: Is NUMA topology relevant?
numactl --hardware
# On multi-socket systems, placement of threads and hot data can matter a lot.

# Step 4: Look at the distribution, not just the mean.
# A large p99/p50 gap usually means the tail needs a separate explanation.
